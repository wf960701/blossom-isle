# 🏗️ 多角色产品开发工作流

> 标准化产品开发流程，将 AI 拆分为多个专业角色（子代理）协作完成一个完整产品。

## 适用场景

- 从 0 到 1 做一个完整产品/功能
- 需要规范化的需求→设计→开发→测试流程
- 任务复杂度高，需要拆分为多个专业领域

## 不适用场景

- 简单的单页面改动（直接改更快）
- Bug 修复（走 debugging 流程）
- 探索性原型验证（走 spike 流程）

## 🔄 整体流程

```
Step 1 ─── 产品经理 ───────────────── 串行，第一步
             │
             ↓ 输出：PRD
             │
Step 2 ─── UI/UX 工程师 ───────────── 串行，需 PRD 定稿
             │
             ↓ 输出：设计规范 + 设计稿
             │
Step 3 ─── 并行执行 ───────────────── 可同时进行
          ┌──────┐    ┌──────┐
          │ 前端  │    │ 后端  │
          │ 工程師│    │ 工程師│
          └──┬───┘    └──┬───┘
             │            │
             ↓            ↓
          ┌───────────────────┐
          │   前后端联调       │
          └────────┬──────────┘
                   │
Step 4 ─── 测试工程师 ──────────────── 串行
                   │
                   ↓
Step 5 ─── 瓜哥验收 + 上线 🎉
```

## 👤 角色定义

### 1. 产品经理

| 项目 | 内容 |
|------|------|
| **职责** | 需求收集、用户故事、功能清单、PRD 撰写 |
| **输入** | 瓜哥的想法和需求描述 |
| **输出** | `prd.md`（完整的 PRD 文档） |
| **子代理任务模板** | 见下方 |
| **关键原则** | 不替用户拍板，每个决策点列出选项让用户确认 |

**子代理任务模板：**

```markdown
你作为产品经理，为以下需求撰写 PRD：
[粘贴用户需求描述]

按照 references/prd-template.md 的格式输出完整的 PRD 文档。
PRD 需要包含：产品概述、用户故事、功能清单、页面路由、数据模型、排期建议。
每个决策点需要给出 2-3 个选项，并注明优缺点，让用户做最终决定。
输出文件：prd.md
```

---

### 2. UI/UX 工程师

| 项目 | 内容 |
|------|------|
| **职责** | 设计规范、组件库、页面高保真设计 |
| **输入** | `prd.md` + 用户确认的设计方向 |
| **输出** | `design/` 目录（规范文档 + 页面设计） |
| **子代理任务模板** | 见下方 |

**子代理任务模板：**

```markdown
你作为 UI/UX 设计师，根据 PRD 文档进行设计：
1. 先阅读 prd.md 了解产品需求
2. 按照 references/design-template.md 输出设计规范
3. 按照 references/page-design-template.md 输出每个页面的高保真 HTML 设计

设计要求：
- 使用 CSS 变量定义色彩系统
- 注意响应式布局
- 考虑边界状态（空状态、加载态、错误态）
- 统一字体、间距、圆角体系

输出文件：
- design/spec.md       → 设计规范
- design/home.html     → 首页设计（示例）
- design/detail.html   → 详情页设计（示例）
```

---

### 3. 前端工程师

| 项目 | 内容 |
|------|------|
| **职责** | 按设计稿实现页面组件、对接后端 API |
| **输入** | `design/` 设计稿 + `prd.md` 需求 |
| **输出** | 前端的页面/组件代码 |
| **子代理任务模板** | 见下方 |

**子代理任务模板：**

```markdown
你作为前端工程师，根据设计稿和 PRD 实现前端页面：
1. 阅读 prd.md 了解业务需求
2. 阅读 design/spec.md 了解设计规范
3. 实现以下页面/组件：[列出页面]

技术要求：
- 使用 [Next.js/React/Vue] + TypeScript
- 使用 [Tailwind/styled-components] 实现样式
- 所有颜色/间距从设计规范 CSS 变量引用
- 组件需覆盖：正常态、空状态、加载态、错误态
- 使用 [Font Awesome/Lucide] 图标库
- 调用后端 API 获取/提交数据

输出：对应的页面/组件代码文件
```

---

### 4. 后端工程师

| 项目 | 内容 |
|------|------|
| **职责** | 数据库设计、API 开发、权限策略 |
| **输入** | `prd.md` |
| **输出** | 数据库迁移脚本 + 相关后端代码 |
| **子代理任务模板** | 见下方 |

**子代理任务模板：**

```markdown
你作为后端工程师，根据 PRD 文档实现后端：
1. 阅读 prd.md 了解数据需求
2. 设计数据表结构
3. 编写数据库迁移脚本（含索引、约束）
4. 编写 RLS 安全策略
5. 提供 API 接口文档

技术要求：
- 数据库：[PostgreSQL / Supabase]
- 表结构需考虑性能和扩展性
- 包含行级安全策略
- API 遵循 RESTful 规范

输出：
- supabase/migrations/NNN_xxx.sql  → 迁移脚本
- 列出所有可用的 API 端点及请求/响应格式
```

---

### 5. 测试工程师

| 项目 | 内容 |
|------|------|
| **职责** | 功能测试、边界测试、回归测试、输出测试报告 |
| **输入** | `prd.md` + 全部开发完成的代码 |
| **输出** | `test-report.md` 测试报告 |
| **子代理任务模板** | 见下方 |

**子代理任务模板：**

```markdown
你作为测试工程师，对已完成的产品进行全量测试：
1. 阅读 prd.md 了解需求
2. 逐条验证功能清单
3. 测试各页面路由是否正常返回 200
4. 测试边界情况：空数据、超长文本、非法输入
5. 测试响应式布局（移动端/桌面端）
6. 测试权限控制是否生效

输出 test-report.md，格式如下：
# 测试报告

## 测试概要
- 测试日期：
- 测试范围：
- 测试结果：通过/部分通过/未通过

## 功能测试
| # | 测试项 | 预期 | 结果 | 备注 |
|---|--------|------|:----:|------|
| 1 | ... | ... | ✅/❌ | ... |

## 边界测试
同上格式

## Bug 清单
| # | 描述 | 严重程度 | 状态 |
|---|------|:--------:|:----:|
| 1 | ... | 高/中/低 | 待修复/已修复 |
```

## 🎬 执行流程

### Step 0: 启动（瓜姐/项目经理）

```
1. 用户提出想法或需求
2. 项目经理确认：走产品开发流程还是直接改？
3. 如走流程 → Step 1
```

### Step 1: 产品经理

```
1. 派发产品经理子代理
2. 子代理产出 prd.md
3. 项目经理把 prd.md 给用户确认
4. 用户确认后 → Step 2
```

### Step 2: UI/UX 工程师

```
1. 派发 UI 子代理（传入 prd.md）
2. 子代理产出 design/ 目录
3. 项目经理展示设计给用户
4. 用户确认后 → Step 3
```

### Step 3: 前后端开发（可并行）

```
1. 派发前端子代理（传入 prd.md + design/）
2. 派发后端子代理（传入 prd.md）
3. 两个子代理并行工作
4. 都完成后，项目经理将前后端代码合并
5. 启动开发服务器验证联调
```

### Step 4: 测试

```
1. 派发测试子代理
2. 子代理执行测试用例
3. 输出 test-report.md
4. 项目经理根据 Bug 清单修复问题
5. 用户确认后 → 上线
```

### Step 5: 上线

```
1. 项目经理确保所有代码已合并到主分支
2. 更新变更日志（log/YYYY-MM.md）
3. 部署上线
4. 通知用户 🎉
```

## 📋 产出物总览

| 步骤 | 产出物 | 谁审批 |
|:----:|--------|:------:|
| ① | `prd.md` | 用户确认 |
| ② | `design/` → spec.md + 页面设计 | 用户确认 |
| ③ | 前端代码 + 后端迁移脚本 | 项目经理 Review |
| ④ | `test-report.md` | 项目经理确认 |
| ⑤ | 变更日志 `log/YYYY-MM.md` | Git 记录 |

---

## ⚠️ 注意事项

1. **每个环节都要用户确认** — 子代理不能替用户拍板
2. **子代理不要太多** — 同一时间最多 3-4 个，否则资源打架
3. **前后端可以并行** — UI 出图后，前端和后端没有强依赖关系
4. **项目经理负责整合** — 子代理各自干活，项目经理负责合并代码、解决冲突
5. **测试不能跳过** — 即使看起来很简单，也要走一遍基本测试
6. **变更日志必须记** — 每次改动都要记入 `log/YYYY-MM.md`
7. **子代理任务要清晰** — 每个子代理的任务模板要包含明确的输入、输出、约束
