# 🏗️ 多角色产品开发工作流

> 标准化产品开发流程，将 AI 拆分为多个专业角色（子代理）协作完成一个完整产品。

> ⚡ 如果已经熟悉这个 skill，可以直接跳到「快速使用」章节直接开用。

---

## 📋 完整产品开发流程（6 阶段）

在拆分子代理之前，先了解完整的传统产品开发流程是怎样的：

### ① 🧠 头脑风暴
**目的：** 把脑子里零碎的想法倒出来，不设限

```
你：我想做一个XX产品
AI：记录所有想法，追问细节
你：这个功能不错，那个也不错
AI：分类整理，圈出核心方向
```
**产出：** 灵感激荡记录 + 功能点子列表
**谁参与：** 你（产品 owner）+ AI（梳理记录）

### ② 📄 PRD（产品需求文档）
**目的：** 把想法变成可执行的文档

一份 PRD 通常包含：

```
1. 产品概述       → 这是什么？给谁用？解决什么问题？
2. 用户故事       → 作为XX用户，我想XX，以便XX
3. 功能清单       → 核心功能、辅助功能、未来功能
4. 页面/路由      → 有哪些页面，各页面做什么
5. 数据模型       → 需要存什么数据，结构如何
6. 非功能需求     → 性能、安全、SEO、响应式等
```

**产出：** 一份完整的 PRD 文档
**谁参与：** 你（确认方向）+ AI（撰写文档）

### ③ 🎨 设计规范
**目的：** 统一视觉语言，保证设计一致性

```
- 色彩系统     → 主色、辅色、中性色、功能色
- 字体系统     → 标题字体、正文字体、字号阶梯
- 间距系统     → 4px 网格、8px 基准
- 组件风格     → 按钮、卡片、输入框、导航
- 图标风格     → 线性/面性、圆角程度
- 动效风格     → 过渡时长、缓动函数
```

**产出：** Design Tokens + 组件规范文档
**谁参与：** AI（出规范）+ 你（确认风格方向）

### ④ 🖼️ 设计图
**目的：** 把规范落地到每个页面

```
可用方式：
- HTML/CSS 原型（AI直出，立刻能看到效果）
- Figma 设计稿（更专业，需手动操作）
- 截图/手绘草图（你画AI实现）
```

**产出：** 每个页面的高保真设计
**谁参与：** AI（出稿）+ 你（Review 确认）

### ⑤ 🔧 前后端开发
**目的：** 按设计图实现功能

```
后端：数据表设计 → API 接口 → 安全策略 → Edge Functions（如有需要）
前端：页面组件 → 状态管理 → API 对接 → 交互动效
```

**产出：** 可运行的代码
**谁参与：** AI（开发）+ 你（验收）

### ⑥ 🧪 测试 & 收尾
**目的：** 确保质量，查漏补缺

```
- 功能测试     → 每个按钮点一遍，每个流程走一遍
- 边界测试     → 空数据、超长文本、错误输入
- 兼容性测试   → 不同浏览器、不同屏幕尺寸
- 性能测试     → 加载速度、图片优化
- 走查验收     → 你当用户用一遍，提修改意见
```

**产出：** 测试报告 + Bug 修复
**谁参与：** AI（测试修复）+ 你（验收签字）

### 传统开发 vs Vibe Coding 对比

| 传统开发 | Vibe Coding |
|---------|------------|
| 先写完整需求文档 | 有个方向就开干 |
| 设计→开发→测试线性推进 | 改改看看，循环迭代 |
| 一次做对 | 快速试错，边做边调 |
| 你定需求我来实现 | 你给感觉我来落地 |

## 👤 角色定义

### 1. 产品经理

| 项目 | 内容 |
|------|------|
| **职责** | 需求收集、用户故事、功能清单、PRD 撰写 |
| **输入** | 瓜哥的想法和需求描述 |
| **输出** | `prd.md`（完整的 PRD 文档） |
| **子代理任务模板** | 见下方 |
| **关键原则** | 不替用户拍板，每个决策点列出选项让用户确认 |

**子代理任务模板：**

```markdown
你作为产品经理，为以下需求撰写 PRD：
[粘贴用户需求描述]

按照 references/prd-template.md 的格式输出完整的 PRD 文档。
PRD 需要包含：产品概述、用户故事、功能清单、页面路由、数据模型、排期建议。
每个决策点需要给出 2-3 个选项，并注明优缺点，让用户做最终决定。
输出文件：prd.md
```

---

### 2. UI/UX 工程师

| 项目 | 内容 |
|------|------|
| **职责** | 设计规范、组件库、页面高保真设计 |
| **输入** | `prd.md` + 用户确认的设计方向 |
| **输出** | `design/` 目录（规范文档 + 页面设计） |
| **子代理任务模板** | 见下方 |

**子代理任务模板：**

```markdown
你作为 UI/UX 设计师，根据 PRD 文档进行设计：
1. 先阅读 prd.md 了解产品需求
2. 按照 references/design-template.md 输出设计规范
3. 按照 references/page-design-template.md 输出每个页面的高保真 HTML 设计

设计要求：
- 使用 CSS 变量定义色彩系统
- 注意响应式布局
- 考虑边界状态（空状态、加载态、错误态）
- 统一字体、间距、圆角体系

输出文件：
- design/spec.md       → 设计规范
- design/home.html     → 首页设计（示例）
- design/detail.html   → 详情页设计（示例）
```

---

### 3. 前端工程师

| 项目 | 内容 |
|------|------|
| **职责** | 按设计稿实现页面组件、对接后端 API |
| **输入** | `design/` 设计稿 + `prd.md` 需求 |
| **输出** | 前端的页面/组件代码 |
| **子代理任务模板** | 见下方 |

**子代理任务模板：**

```markdown
你作为前端工程师，根据设计稿和 PRD 实现前端页面：
1. 阅读 prd.md 了解业务需求
2. 阅读 design/spec.md 了解设计规范
3. 实现以下页面/组件：[列出页面]

技术要求：
- 使用 [Next.js/React/Vue] + TypeScript
- 使用 [Tailwind/styled-components] 实现样式
- 所有颜色/间距从设计规范 CSS 变量引用
- 组件需覆盖：正常态、空状态、加载态、错误态
- 使用 [Font Awesome/Lucide] 图标库
- 调用后端 API 获取/提交数据

输出：对应的页面/组件代码文件
```

---

### 4. 后端工程师

| 项目 | 内容 |
|------|------|
| **职责** | 数据库设计、API 开发、权限策略 |
| **输入** | `prd.md` |
| **输出** | 数据库迁移脚本 + 相关后端代码 |
| **子代理任务模板** | 见下方 |

**子代理任务模板：**

```markdown
你作为后端工程师，根据 PRD 文档实现后端：
1. 阅读 prd.md 了解数据需求
2. 设计数据表结构
3. 编写数据库迁移脚本（含索引、约束）
4. 编写 RLS 安全策略
5. 提供 API 接口文档

技术要求：
- 数据库：[PostgreSQL / Supabase]
- 表结构需考虑性能和扩展性
- 包含行级安全策略
- API 遵循 RESTful 规范

输出：
- supabase/migrations/NNN_xxx.sql  → 迁移脚本
- 列出所有可用的 API 端点及请求/响应格式
```

---

### 5. 测试工程师

| 项目 | 内容 |
|------|------|
| **职责** | 功能测试、边界测试、回归测试、输出测试报告 |
| **输入** | `prd.md` + 全部开发完成的代码 |
| **输出** | `test-report.md` 测试报告 |
| **子代理任务模板** | 见下方 |

**子代理任务模板：**

```markdown
你作为测试工程师，对已完成的产品进行全量测试：
1. 阅读 prd.md 了解需求
2. 逐条验证功能清单
3. 测试各页面路由是否正常返回 200
4. 测试边界情况：空数据、超长文本、非法输入
5. 测试响应式布局（移动端/桌面端）
6. 测试权限控制是否生效

输出 test-report.md，格式如下：
# 测试报告

## 测试概要
- 测试日期：
- 测试范围：
- 测试结果：通过/部分通过/未通过

## 功能测试
| # | 测试项 | 预期 | 结果 | 备注 |
|---|--------|------|:----:|------|
| 1 | ... | ... | ✅/❌ | ... |

## 边界测试
同上格式

## Bug 清单
| # | 描述 | 严重程度 | 状态 |
|---|------|:--------:|:----:|
| 1 | ... | 高/中/低 | 待修复/已修复 |
```

## 🎬 执行流程

### Step 0: 启动（瓜姐/项目经理）

```
1. 用户提出想法或需求
2. 项目经理确认：走产品开发流程还是直接改？
3. 如走流程 → Step 1
```

### Step 1: 产品经理

```
1. 派发产品经理子代理
2. 子代理产出 prd.md
3. 项目经理把 prd.md 给用户确认
4. 用户确认后 → Step 2
```

### Step 2: UI/UX 工程师

```
1. 派发 UI 子代理（传入 prd.md）
2. 子代理产出 design/ 目录
3. 项目经理展示设计给用户
4. 用户确认后 → Step 3
```

### Step 3: 前后端开发（可并行）

```
1. 派发前端子代理（传入 prd.md + design/）
2. 派发后端子代理（传入 prd.md）
3. 两个子代理并行工作
4. 都完成后，项目经理将前后端代码合并
5. 启动开发服务器验证联调
```

### Step 4: 测试

```
1. 派发测试子代理
2. 子代理执行测试用例
3. 输出 test-report.md
4. 项目经理根据 Bug 清单修复问题
5. 用户确认后 → 上线
```

### Step 5: 上线

```
1. 项目经理确保所有代码已合并到主分支
2. 更新变更日志（log/YYYY-MM.md）
3. 部署上线
4. 通知用户 🎉
```

```
  - 标签选择 React 组件            - 确保 article_tags 表
  - 多选 UI + 搜索过滤             - 增删标签关联 API

  - 粉色调样式                     - 接口文档
     └──────────────┬───────────────┘
                    ↓

  ✅ 可选多个标签
  ✅ 保存后重新打开标签正确回显
  ✅ 空标签状态显示友好提示
  ✅ 删除文章同步删除关联
                    ↓
用户验收 → 合并 → 部署上线 🎉
```

## ⚠️ 注意事项

### 关键约束

| # | 规则 | 说明 |
|:-:|:-----|:------|
| 1 | **每环节用户确认** | 子代理不能替用户拍板，PRD/设计必须给用户看过 |
| 2 | **子代理上限 3-4 个** | 同时派发太多会资源打架，影响质量 |
| 3 | **前后端可并行** | UI 出图后前端和后端没有强依赖关系 |
| 4 | **项目经理整合** | 子代理各自产出，项目经理负责合并代码、解决冲突 |
| 5 | **测试不能跳过** | 即使看起来简单的改动也要走基本测试 |
| 6 | **变更日志必记** | 每次改动记入 `log/YYYY-MM.md`，按月分文件 |
| 7 | **任务模板要清晰** | 每个子代理的任务包含：输入文件、输出文件、技术要求、约束条件 |

### 常见问题

**子代理之间代码冲突怎么办？**
> 项目经理负责合并，前端和后端代码通常在不同目录（前端 `src/app/`，后端 `supabase/`），冲突概率低。

**用户中途改了需求怎么办？**
> 回到 Step 1 产品经理更新 prd.md，然后从受影响环节重新走。变动小可直接改，变动大需重新确认。

**子代理产出的质量如何保证？**
> 项目经理 review 所有产出。测试工程师做最后一道把关。每个环节用户确认就是质量控制点。

**技能如何迁移给其他 AI？**
> 其他 AI 只需读取 `SKILL.md` 和 `references/` 下的模板，按角色任务模板派发子代理即可。不依赖特定平台或工具。
